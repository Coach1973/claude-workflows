永遠選擇最節省 token 的方案，在執行前先評估消耗，高風險操作必須先告知用戶。

**Why:** 用戶因為多次 token 浪費事件損失嚴重，每次超支都會導致額度提前用盡、工作中斷到下次重置，嚴重影響工作效率。這個規則被提醒過多次，沒有任何藉口再違反。

## 🚨 2026-04-12 強化守則：探索性行為必須先請示

**事件**：克勞德在執行 NotebookLM 搬家任務時，20 分鐘內消耗了用戶 43% 的週配額。根本原因：用大量探索性的小工具呼叫（讀原始碼、查方法簽名、測試腳本）來「摸索」API，而非直接寫腳本。

**新增守則**：
- **「探索性行為」定義**：目的是摸清工具/API/格式，而非直接完成任務的工具呼叫
- **禁止規則**：除非用戶明確授權，探索性行為嚴禁超過 3 次工具呼叫
- **正確做法**：不確定 API 時，先告訴用戶「我需要做 X 次探索性查詢，預估消耗 Y，確認後才執行」
- **緊急做法**：如果已經開始探索，立刻停下來，直接寫最可能正確的腳本跑，錯了再說

## ⛔ 高危操作黑名單（執行前必須警告）

| 操作 | 風險原因 | 替代做法 |
|------|---------|---------|
| `notebook_list`（107個筆記本） | 回傳超過 15,000 tokens JSON | 直接用記憶裡的 notebook ID，不查 |
| `WebFetch × N`（N > 5） | N倍 token 爆炸 | 改用 PowerShell 腳本 |
| 多 Agent 並行 | 每個 Agent 獨立消耗 | 必須先告知預估費用 |
| 讀取大型 JSON / XML | 內容送進 context | 腳本解析，只傳結果 |
| 跨對話 context 繼承 | 摘要可能超大 | 一任務一對話，避免 context 爆掉 |

## How to apply

1. **執行任何工具前，先心算**：這個工具回傳多少資料？送進 context 多少 tokens？
2. **超過 5 萬 tokens 估計** → 必須先告知用戶，等確認再執行（YouTube 分析、大量 WebFetch、多 Agent 等任務）
3. **5,000～50,000 tokens** → 可執行，但完成後主動回報消耗
3. **有記憶就用記憶**：notebook ID、帳號、規則都已記錄，不重複查詢
4. **「重複性資料處理」永遠用腳本**，Claude 只負責理解與判斷
5. 測試先用最小範圍（1個）驗證，再擴大
6. 遇到不確定方案，先說明兩種做法的 token 差異，讓用戶選擇
